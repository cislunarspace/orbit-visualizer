## Agent Brief

**Category:** enhancement
**Summary:** Create one Satellite catalog interface that drives both built-in TLE selection and the default Orbit computation input

**Current behavior:**
Built-in `Satellite` knowledge is duplicated. Orbit computation has its own default `Satellite`, while the GUI owns a separate preset table for TLE input. Updating built-in satellite data requires changing more than one module, and the two paths can drift.

**Desired behavior:**
One deep `Satellite` catalog module should expose the built-in `Satellite` values, the default `Satellite`, and any lookup needed by callers. Both the GUI preset flow and Orbit computation should consume that same interface so one built-in `Satellite` produces one consistent end-to-end behavior.

**Key interfaces:**
- `Satellite` - the shared domain value for satellite identity plus TLE lines
- A catalog interface that lists built-in `Satellite` values and returns the default `Satellite`
- GUI preset selection should resolve to a `Satellite` from the catalog rather than a duplicate local mapping
- Orbit computation callers should request the default `Satellite` from the catalog rather than owning separate default knowledge

**Acceptance criteria:**
- [ ] A shared interface exposes built-in `Satellite` values and the default `Satellite`
- [ ] Built-in TLE data lives in one place only
- [ ] The GUI preset flow and Orbit computation both consume the same catalog interface
- [ ] Tests verify that a built-in `Satellite` flows consistently through both entry points

**Out of scope:**
- Building the orbit request module
- Refactoring CZML formatting
- Adding new external satellite data sources
