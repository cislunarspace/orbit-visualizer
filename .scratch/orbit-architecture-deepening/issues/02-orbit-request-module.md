## Agent Brief

**Category:** enhancement
**Summary:** Introduce one orbit request interface that carries user-selected Orbit computation inputs from the GUI to the API seam

**Current behavior:**
The GUI exposes ad hoc dicts and tuples for orbit options, server settings, and TLE input, while the API path separately decides how to interpret request inputs. The real workflow is spread across callers instead of concentrated behind one module interface.

**Desired behavior:**
One orbit request module should represent the user-selectable inputs for Orbit computation: orbit types, time span, step, base time, radius override, and chosen `Satellite` or custom TLE. GUI code should assemble that module, and the API path should consume it, so a single request shape drives end-to-end behavior.

**Key interfaces:**
- `OrbitRequest` - a domain-facing request value that captures orbit type selection, timing, radius override, and `Satellite` / custom TLE choice
- Validation rules for accepted orbit types, timing fields, and mutually compatible request inputs
- GUI adapters that translate widget state into an `OrbitRequest`
- API-side logic that consumes an `OrbitRequest` to choose Orbit computation adapters

**Acceptance criteria:**
- [ ] One interface represents the full Orbit computation request shape
- [ ] Validation for request inputs lives behind that interface instead of being spread across callers
- [ ] GUI code produces an `OrbitRequest` rather than only ad hoc dicts and tuples
- [ ] The API path consumes the shared request interface to decide which orbit paths run
- [ ] Tests verify one request flowing from UI state to API-facing behavior

**Out of scope:**
- Changing orbit mathematics
- Deepening the TLE adapter itself
- Changing CZML styling or browser presentation
